## 최상위 — 2개 활동 카테고리 + 주차장 + 운영 시스템

활동을 **데이터 성격**으로 두 카테고리로 가른다. 분류 안 되는 건 주차장.
활동이 아니라 **이 맥에서 실제 가동 중인 시스템**은 별도 카테고리에 둔다.

| 카테고리 | 정의 |
|----------|------|
| **가-process** (생애주기형) | 데이터가 **idea→develop→result→share**로 흐르는 작업. 시작·진행·완료·보관의 생애주기가 있다. |
| **나-routine** (반복업무형) | **생애주기 없이 반복**되는 업무. "반복 루틴(양식·작업공간)"과 "그 결과물"만 있다. |
| **다-parking** (주차장) | 어디에도 분류 안 되는 것 임시 보관 (평평). |
| **라-custom-system** (운영 시스템) | 이 맥의 **시스템 운용 효율을 위해 개발되어 실제 가동 중인** 시스템 파일. 예) 자체 개발 MCP·스크립트·자동화 도구. (평평 — 시스템별 폴더) |

**가·나** 카테고리는 다시 **도메인** 3개로 나뉜다. (`다`·`라`는 도메인 없이 평평)

| 도메인 | 의미 |
|--------|------|
| **1-research** (리서치) | 업무를 보조하거나 새로 개발하기 위한 프로젝트. **장기적·지속적** 성격을 띤다. 여기서 실행되는 프로세스는 조직 내 **장기적·시스템적 개선**을 향한다. (개발앱·지식·R&D) |
| **2-business** (비즈니스) | 회사 조직에서 **지금 일어나고 있는 직접적·단기적** 업무. **현재성·현장성**이 핵심이다. (수주·계약·현장) |
| **3-personal** (퍼스날) | 업무·조직과 **무관한 개인**의 사생활·취미·사회경제 전반의 일. |

> ⚠️ 핵심 구분: "개발한 앱"은 비즈니스가 아니라 **research**(장기 개발)다. business는 "지금 진행 중인 회사 업무" 그 자체.
>
> ⚠️ **`라-custom-system` vs `1-research`**: research는 **연구·개발하는 것**(탐색·실험·지식), 라는 이 맥북에서 **실제로 돌아가는 시스템** 그 자체(운영). 같은 자체 개발 MCP라도 — 연구·제작 중이면 `가-process/1-research`, 실가동에 투입되면 `라-custom-system`.

## Process — 생애주기 5단계

| 단계 | 의미 |
|------|------|
| **reference** | 업무에 필요해 조사한 자료·외부 공급자료 등 **프로세스 전반에 영향을 줄 수 있는 각종 자료가 창고처럼 쌓이는 곳**. (파이프라인 밖, 항상 곁에 둠) |
| **inbox** | 프로젝트를 처음 생성해 **아직 방향성이 정해지지 않고 초안 수준**의 업무가 일어나는 곳. 새로 만드는 것이든, 기존 프로젝트(action·result 등)가 계획 변경으로 **다시 돌아온 재유입**이든 — **new/renewal 구분 없이 하나의 인박스**로 다룬다(프로젝트 폴더를 인박스 직속에 둔다). |
| **action** | inbox에서 방향성·초안이 잡혀 **고도화되어 가는 업무 — 완성 후 테스트까지**의 과정이 들어있는 곳 |
| **result** | action에서 고도화·테스트까지 거쳐 **안정화되어 일상화된** 업무/프로젝트가 있는 곳 |
| **share** | result에서 충분히 검증돼 그 시스템을 **템플릿으로 추출해 공유**하거나 **장기 저장소로 이동**하는 곳. 템플릿 파일, 또는 장기저장 링크 등의 **인덱스로 체계적으로 정리**된 폴더 |

## Routine — 작업공간 + 푸시 모델

| 버킷 | 의미 |
|------|------|
| **routines/`<루틴명>`** | 반복 작업의 **작업공간**. 양식·템플릿 + 매 사이클 작업 데이터(수집·기록·정제)가 여기서 처리됨. |
| **outputs/`<루틴명>`** | **푸시된 결과 누적**. routines와 같은 루틴명으로 대응. |

- **푸시(커밋)**: "푸시해/아웃풋해/결과 이동해" → 정제 결과를 `outputs/<루틴명>/`로 보낸다(날짜 부여).
- 푸시 후 작업공간 정리는 **루틴별 설정**대로: 양식만 남기거나(업무일지형) / 비움(뉴스수집형).
- 출력 형태는 루틴 설계 자유(데이터만 / 채워진 양식). 상세 → [AGENTS.md](./AGENTS.md) Routine 푸시 프로토콜.

## 넘버링 — Johnny-Decimal (계층마다 자리 추가)

```
카테고리 폴더   가-process · 나-routine · 다-parking · 라-custom-system
도메인 폴더(1자리)   1=research 2=business 3=personal      예) 가-process/1-research · 나-routine/1-research
Process 단계코드 = [도메인][단계] 2자리
   단계  0=reference 1=inbox 2=action 3=result 4=share   (inbox는 하위 구분 없이 프로젝트 직속)
Routine 버킷코드 = [도메인][버킷] 2자리
   버킷  0=routines 1=outputs
```
- 가-process `10`=research·reference · `12`=action · `31`=personal·inbox · 나-routine `10`=research·routines
- 프로젝트/결과 폴더 = `YYYY_MM_DD_이름`(생성일 순). 루틴 폴더 = 기능명(날짜 없음).
- 카테고리(가/나/다/라)가 **부모로 구분**하므로, 같은 숫자코드가 process·routine 양쪽에 있을 수 있다(부모로 식별).
- `다-parking`·`라-custom-system`은 도메인·단계 코드를 쓰지 않는다(평평). 라는 시스템 이름으로 폴더를 만든다.

## 배치 규칙

0. **이 맥에서 실제 가동되는 시스템인가?** (자체 MCP·스크립트 등) → `라-custom-system`. 아직 연구·개발 중이면 아래로.
1. **반복업무인가, 생애주기 작업인가?** 반복 → `나-routine`, 흐름 있는 작업 → `가-process`.
2. **어느 도메인?** 개발·연구·지식 → research, 회사운영 → business, 사적 → personal.
3. **Process면 어느 단계?** 참고용→reference, 아이디어→inbox, 진행→action, 완료→result, 배포/보관→share.
4. **애매하면** → `다-parking`. (마이그레이션을 막지 않기 위해)
5. 빈 분류함을 미리 만들지 않는다. 대용량 미디어는 inbox에 적체시키지 않는다.
